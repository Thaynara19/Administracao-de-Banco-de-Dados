# 📘 Recuperação de Arquivo Utilizando Git

Hoje enfrentei um problema durante um `git pull --rebase`.

Durante a resolução de um conflito, um dos meus arquivos SQL acabou ficando praticamente vazio, contendo apenas uma linha de código. Como era um exercício em desenvolvimento, pensei que havia perdido todo o conteúdo.

---

## ⚠️ O problema

Durante o rebase ocorreram conflitos entre as versões local e remota do arquivo.

O arquivo afetado foi:

```text
PRATICANDO_EXERCÍCIOS.sql
```

Ao abrir o arquivo, percebi que praticamente todo o código havia desaparecido.

---

## 🔎 Investigação

Antes de tomar qualquer decisão, utilizei alguns comandos do Git para entender o que estava acontecendo.

```bash
git status
```

Verifiquei que o rebase ainda estava em andamento.

Depois consultei o histórico dos commits:

```bash
git log --oneline
```

E utilizei o reflog para localizar onde o histórico havia sido alterado:

```bash
git reflog
```

Também consultei versões anteriores do arquivo utilizando:

```bash
git show <hash_do_commit>:caminho/do/arquivo
```

---

## ✅ Recuperação

Após localizar a versão correta do arquivo, consegui recuperar todo o conteúdo que havia sido perdido.

Em seguida:

- resolvi os conflitos;
- finalizei o rebase;
- mantive o histórico organizado.

---

## 📚 O que aprendi

Hoje entendi melhor como o Git trabalha durante um **rebase**.

Também aprendi que:

- conflitos não significam perda definitiva do código;
- o histórico do Git permite recuperar versões anteriores;
- comandos como `git status`, `git log`, `git reflog` e `git show` são extremamente úteis para investigação.

---

## 💡 Evolução

Esse exercício me mostrou a importância de conhecer ferramentas de recuperação de código.

Assim como um DBA utiliza backups e logs para recuperar bancos de dados, um desenvolvedor utiliza o histórico do Git para recuperar versões anteriores dos arquivos.

Cada ferramenta protege um tipo diferente de informação, mas ambas fazem parte das boas práticas de desenvolvimento.
